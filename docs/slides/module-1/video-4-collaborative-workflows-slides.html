Anatomy of a Good Pull Request

‚úÖ Clear Title:
"Add support for GeoJSON export in QGIS plugin"
Description Template:

Adds CSV export functionality to spatial analysis results

## Motivation
Users requested ability to export for Excel (#42)

## Changes
- Added export_to_csv() function in utils.py
- Updated UI with "Export" button
- Added tests for CSV generation

## Testing
- Manual testing with sample datasets
- All existing tests pass
- New tests added for CSV export

Good PRs get merged; poor PRs get ignored

A good PR has a clear title, detailed description explaining what, why, and how, shows testing, and includes screenshots if relevant. This makes review easier and increases chances of acceptance.

Code Review Best Practices

As a Reviewer:

  • ‚úÖ Be respectful and constructive
  • ‚úÖ Focus on code, not person
  • ‚úÖ Explain "why" when requesting changes
  • ‚úÖ Acknowledge good work
  • ‚úÖ Suggest alternatives, don't just criticize
  • ‚úÖ Review promptly (contributors are waiting)
‚úÖ Good Review Comment:

‚ùå Bad Review Comment:

As a reviewer, be respectful and constructive. Focus on code, not the person. Explain why you're suggesting changes. Acknowledge good work. The goal is collaborative teaching, not criticism.

Understanding Merge Conflicts

ü§î What Are Merge Conflicts?

  • Two people edited the same lines of code
  • Git can't automatically merge
  • You must manually decide which changes to keep
‚ö†Ô∏è Common Scenario:
You're working on feature branch ‚Üí Meanwhile, main branch updated ‚Üí Your branch now conflicts with main

üí° The Good News: Conflicts are normal! Resolution is a learned skill.

Merge conflicts happen when two people edit the same lines. Git can't automatically decide which to keep. This is completely normal‚Äînot a disaster. Resolution is a learned skill that becomes easy with practice.

Resolving Merge Conflicts (Web UI)

‚ö†Ô∏è When Conflicts Occur: GitHub shows "This branch has conflicts that must be resolved"
1. Click "Resolve conflicts" button on the Pull Request page
GitHub opens web-based conflict editor
2. GitHub shows conflict markers: 
<<<<<<< your-feature-branch (your changes)
result = calculate_distance(point1, point2)
=======
result = compute_spatial_distance(point1, point2)
>>>>>>> main (their changes)
3. Edit in Web Editor:
‚Ä¢ Delete conflict markers (<<<<<<<, =======, >>>>>>>)
‚Ä¢ Keep your changes, their changes, or blend both
‚Ä¢ Result: result = compute_spatial_distance(point1, point2)
4. Mark as resolved: Click "Mark as resolved" button
5. Commit merge: Click "Commit merge" button
‚úÖ Conflict resolved! PR can now be merged
GitHub's web UI makes conflict resolution easier. When conflicts occur, click Resolve Conflicts button. GitHub shows you the conflict markers. Edit directly in the browser‚Äîremove markers, choose which code to keep. Mark as resolved, commit the merge. All done in the web UI‚Äîno command line needed.

Preventing Merge Conflicts

  • üîÑ Sync with main frequently
    Don't let your branch drift too far
  • ‚ö° Keep branches short-lived
    Merge quickly, don't leave branches open for weeks
  • üí¨ Communicate with team
    "I'm working on the auth module"
  • üéØ Small, focused changes
    Smaller PRs = fewer conflicts
  • üë• Divide work clearly
    Different developers, different files when possible

Prevention is easier than resolution!

Prevent conflicts by syncing frequently, keeping branches short-lived, communicating with your team, making small focused changes, and dividing work clearly. Prevention is much easier than resolution.
